    Continuous Integration / Continuous Deployment

    Setting up CI/CD is an always ongoing process, the base has to be done intially, but the process will always grow and there will always be something new to add/setup/define as the project evolves.

    A CI/CD usually consist of several steps, a code quality checker like (es-lint, scss-lint), a test suite be that Unit Tests, E2E Tests a build tool and then and the CD is the automatic deployment tool.

    Unit Tests

    Fast Paced start-ups don't give you the possiblity to write too many tests. Enterprise projects are much better at this usually as they can afford the extra time, and cannot afford the potential risk of untested code.

  • JS Frameworks

    React

    Love it. Main togo framework. I do consider that React is not always the way to go. It all depends on the scope and goal of your project.

    Love the Redux approach and the scallability, it requires a lot of boilerplating but on scale manner it pays-off.

    A massive advantage of React is the React Native framework. It just brings the mobile development closer to Web Developers, this allows everybody to get an entry point into Mobile Development, which is purely magical.

    Vue

    I find it extremly popular now-a-days but I miss a lot of stuff there. And I'm not a big fan of the mixed HTML Angular like. In my opinion it is great for small SPAs but I'd go with React for a large APP.

    jQuery

    I still think it was and is a great library for simple web sites, which don't require a lot of js work. It can't really keep upwith how web apps evolved now-a-days but it's a must have in your tech backpack.

    Google Closure Script

    It's a tool that basically mix jQuery and Modern Web Development. You build components, but you bind them to existing DOM, basically trying to keep up the Single Responsibility Principle yet make everything modular.

    It is a pain to configure it, because it uses a .jar file to compile js, but it does an amazing job.
